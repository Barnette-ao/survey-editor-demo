## Undo / Redo 设计前思考清单

### ① undo / redo 操作的对象是什么？
- 是整个 `runningStorage` 的状态？
- 还是某一类操作（如：新增题目 / 删除题目 / 修改属性）？
- Design 页 和 JSON Editor 是否共用同一套 undo / redo 机制？
- 是否存在不同粒度的回退需求？

答：
我想要Design页面能够使用某一类操作的方式作为操作对象，
而JSON Editor使用整个状态的方式作为操作对象
---

### ② undo 的边界在哪里？
- 切换 tab 是否需要清空 undo 栈？
- 手动保存（save）是否是一个 undo 边界？
- 自动保存是否应该影响 undo 栈？
- 路由切换是否意味着 undo 失效？
- 页面刷新后的 undo 是否需要恢复？

答：
- 切换tab时，不需要清空undo栈。
原因：undo和redo存储的数据，其实可以划分为待确定态/确定态，
一个频繁的用户操作就是操作一下编辑器页或者JSON编辑页，然后切换预览层看效果，如果不满意，当然会使用undo/redo功能，清空undo栈意味着数据进入确定态，没办法在反悔了，这与上述用户操作预期冲突，所以切换tab时，不需要清空undo栈。

- 手动保存是undo的一个边界，此时数据应该进入确定态。

- 自动保存应该兼容undo栈。
现在实现的自动保存是直接调用领域层的save, 所以这种自动保存是直接预设所有的数据都是确定态没有待确定态。因此现阶段自动保存功能与undo功能冲突。
解决办法是，在领域层之上再加一层草稿层。
自动保存是保存的草稿层，undo/redo也是操作的草稿层，草稿层的数据全部都是待确定态
点击手动保存按钮或者退出编辑按钮的时候，进入确定态。

- 路由切换等同于手动保存或者退出编辑按钮，即意味着undo失效

- 页面刷新应该丢失undo。
原因：MVP 阶段：页面刷新 = undo 丢失
要支持刷新恢复 undo：undo 栈持久化，版本兼容，操作回放 or 状态快照恢复
这些都是优化内容。现在不做持久化，制作显示状态绑定。
设计上可以留住扩展点，但不要在第一版就马上实现
---

### ③ 自动保存 + undo 是否会产生冲突？
- undo 回退到某个状态时，是否立即触发保存？
- 如果不立即保存，何时保存？
- undo / redo 操作本身是否应该被记录进 undo 栈？
- 自动保存失败时，undo 状态如何处理？

答
- 该问题的答案参考自动保存应该兼容undo栈。
- 该问题的答案参考自动保存应该兼容undo栈。
- undo / redo 操作本身不应该被记录进 undo 栈，先实现的应该简单一点，不要加这么多的数据转换规则
- undo栈和redo栈的结构，此时以Design组件为背景，undo栈中存入的元素是操作后的元素，表示操作的顺序，然后弹出undo的元素，会加入redo栈.
- 自动保存失败时，undo不会失败。
undo 栈是内存中的用户操作历史，autosave 是草稿持久化尝试
持久化尝试失败不会影响用户操作历史
持久化尝试失败：
1. undo不回滚
2. 标记草稿为dirty或者unsaved
3. UI提示保存失败
下一次autosave或手动save再尝试
目的：避免网络问题导致的用户操作丢失
---

### ④ undo 是“用户意图”还是“数据回退”？
- 输入框连续输入（多次变更）是否需要合并？
- 拖拽排序是否应视为一次原子操作？
- 是否需要对操作进行分组（batch / transaction）？
- 是否区分“瞬时变更”和“结构性变更”？

答
- 输入框连续变更用debounce包裹记录undo的操作。所以不用刻意的合并
- 可以把一次拖拽视为一次原子操作，但是同样要用debounce包裹，并且间隔时间要在3秒
- 是否需要对操作进行分组，这是属于优化的内容，先做出来一个可用的版本再说这件事，所以暂时不会分组
- 是否区分“瞬时变更”和“结构性变更”？这个问题答案同上，不会区分，这是后续的优化内容

---

### ⑤ undo / redo 属于哪一层？
- 领域层（数据模型）？
- 编辑器上下文层（Editor 容器）？
- 作为一个独立的 composable？
- 是否需要与 UI 状态解耦？

答
- 属于草稿层
- 领域层 - 草稿层 - 编辑上下文层，从上到下分层即如此
- 不知道，没想好。
- 位于草稿层，会有一定的地方与上下文层耦合，UI状态输入上下文层，所以可能需要。

### ⑥ 最小可行版本（MVP）应满足什么？
- undo 栈 / redo 栈的基本结构
- 明确什么时机入栈
- 明确什么时机清空 redo 栈
- 明确哪些操作 **不进入** undo 栈
