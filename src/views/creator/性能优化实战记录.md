### 遇到的问题和解决方案

**问题1：导出和引用方式不匹配**

```javascript
// ❌ 错误的导出方式
export default mockQuestionnaireAPI  // 但是 mockQuestionnaireAPI 没有定义

// ❌ 错误的引用方式
import { mockQuestionnaireAPI } from '@/mock/questionnaire.js'  // 尝试解构导入

// 错误信息：
// Uncaught SyntaxError: The requested module does not provide an export named 'mockQuestionnaireAPI'
```

**解决方案**：

```javascript
// ✅ 正确的导出方式（命名导出）
export const mockQuestionnaireAPI = {
  getQuestionnaireById(id) { ... },
  updateQuestionnaire(id, data) { ... }
}

// ✅ 正确的引用方式
import { mockQuestionnaireAPI } from '@/mock/questionnaire.js'
```

**关键学习点**：
- `export default` 用于默认导出，引用时用 `import xxx from 'xxx'`
- `export const` 用于命名导出，引用时用 `import { xxx } from 'xxx'`
- 导出的变量必须先定义，不能导出未定义的变量

拟列表与拖拽功能冲突分析及组件懒加载优化方案

### 问题背景
用户发现使用虚拟列表（vue-virtual-scroller）后，题目间距消失，出现空白区域，且与 Sortable.js 拖拽功能产生严重冲突。

### 虚拟列表问题分析

#### 1. 样式问题
- 虚拟列表会重置或覆盖原有的 margin/padding 设置
- DynamicScrollerItem 会包裹每个题目，可能重置子元素的 margin
- 为了性能优化，虚拟列表会预留缓冲区域，动态计算和调整容器高度

#### 2. 与 Sortable.js 的冲突
**严重冲突点：**
- DOM 元素动态创建/销毁：拖拽过程中元素可能被回收
- 改变 DOM 结构：Sortable 绑定的元素可能不存在  
- 索引计算错误：虚拟列表的索引与实际数据索引不匹配
- 拖拽句柄失效：.drag-handle 可能在滚动时被销毁
- 跨页拖拽失效：目标容器可能不在 DOM 中

#### 3. vue-draggable-next 同样存在冲突
无论是 Sortable.js 还是 vue-draggable-next，都会与虚拟滚动产生相同的根本性冲突：
- 虚拟滚动会动态创建/销毁 DOM 元素
- 拖拽库需要稳定的 DOM 结构来跟踪元素
- 索引计算混乱：虚拟滚动的可见索引 ≠ 实际数据索引

### 最终选择：组件内部懒加载优化

#### 核心思路
不改变整体架构，从组件内部优化：
- 保持现有的页面结构和拖拽功能
- 只有选中的题目才完整渲染内容区域
- 未选中的题目只显示标题和操作按钮
- 选项列表、编辑器等重型组件异步加载

#### 实现方案

**1. BaseQuestion 组件优化**
```vue
<template>
  <div class="question-container">
    <!-- 题目头部（始终显示） -->
    <div class="question-header">
      <!-- 标题和操作按钮 -->
    </div>

    <!-- 异步加载的内容区域 -->
    <template v-if="shouldLoadContent">
      <div class="options-container">
        <slot name="options"></slot>
      </div>
    </template>
    
    <!-- 加载占位符 -->
    <div v-else-if="isLoading" class="content-loading">
      <el-skeleton :rows="3" animated />
    </div>
  </div>
</template>

<script>
const shouldLoadContent = computed(() => {
  if (!props.lazyLoad) return true
  return contentLoaded.value || selected.value
})

// 监听选中状态，异步加载内容
watch(selected, async (newSelected) => {
  if (newSelected && !contentLoaded.value && props.lazyLoad) {
    isLoading.value = true
    await nextTick()
    setTimeout(() => {
      contentLoaded.value = true
      isLoading.value = false
    }, 100)
  }
})
</script>
```

**2. RadioGroup 组件优化**
```vue
<template>
  <base-question :lazy-load="true">
    <template #options>
      <AsyncOptionsContainer v-if="shouldRenderOptions" />
      <div v-else class="options-placeholder">
        <span>{{ element.choices?.length || 0 }} 个选项</span>
      </div>
    </template>
  </base-question>
</template>

<script>
// 异步组件定义
const AsyncOptionsContainer = defineAsyncComponent(() => 
  import('./components/OptionsContainer.vue')
)

// 监听选中状态，按需加载组件
watch(isSelected, (selected) => {
  if (selected) {
    shouldRenderOptions.value = true
  }
})
</script>
```

#### 性能优化效果

**优化前：**
- 100个题目 = 100个完整组件 + 400个选项组件 + 100个编辑器 = 600+组件实例
- DOM节点: 10000+
- 初始渲染时间: 长

**优化后：**
- 100个题目 = 100个基础组件 + 1个完整组件（选中的） = 101个组件实例  
- DOM节点: 1000+
- 初始渲染时间: 大幅减少

#### 优势
1. **保持所有现有功能**：拖拽、页面管理等完全不受影响
2. **显著性能提升**：减少90%的组件实例和DOM节点
3. **用户体验优秀**：快速浏览，按需加载
4. **实现简单**：不破坏现有架构，渐进式优化
5. **完全向后兼容**：可以通过 lazyLoad 属性控制是否启用

### 结论
经过深入分析，虚拟列表方案由于与复杂的拖拽功能存在根本性冲突而被放弃。最终选择组件内部懒加载方案，既能大幅提升性能，又能保持所有现有功能的完整性。这是在复杂业务场景下的最优解决方案。
